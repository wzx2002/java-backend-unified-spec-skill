# Java 后端安全基线规范

## 20. 密钥与配置安全

### 20.1 基本原则

- 密钥、令牌、证书、验签密钥、数据库密码、消息队列密码、对象存储密钥、第三方 `AK/SK` 不允许硬编码在源码中
- 禁止把生产环境密钥提交到 Git、打进镜像、写入默认配置文件、写入示例代码
- 配置按环境隔离，至少区分 `local / dev / test / staging / prod`
- 敏感配置统一通过环境变量、配置中心、密钥管理系统或部署平台注入
- 敏感配置的读取、刷新、轮换方式必须有明确归属，不允许“谁用到谁自己临时搞一套”

### 20.2 命名与分层

- 敏感配置类统一放 `infrastructure/config` 或对应三方 `client/properties`
- 配置类命名使用 `XxxProperties`
- 与第三方厂商相关的密钥，优先跟厂商客户端放在同一业务边界下，例如 `infrastructure/client/payment/PaymentProperties`
- 业务层、领域层不直接感知原始密钥值，只依赖封装后的客户端能力

### 20.3 脱敏与展示

- 日志、监控、异常消息、接口响应中禁止输出完整密钥、Token、验证码、银行卡号、身份证号、手机号、邮箱、签名串、回调原文
- 必须展示时，只允许输出脱敏值，例如保留前 3 后 4
- 故障排查时如确需查看原文，必须走受控审计手段，不能依赖临时打印日志

### 20.4 轮换与失效

- 关键密钥必须支持轮换，不允许设计成只能重启服务和全量改代码才能替换
- JWT 密钥、支付验签密钥、对象存储密钥、短信签名密钥等必须明确失效策略和切换窗口
- 密钥轮换期间需要双 Key 兼容时，必须明确主 Key、旧 Key、失效时间和回收步骤

## 21. 输入输出与边界安全

### 21.1 输入校验

- 所有外部输入都视为不可信，包括 HTTP 参数、Header、Cookie、MQ 消息、定时任务参数、Webhook 回调、文件内容、第三方同步数据
- `web / interfaces` 层必须先完成格式校验、长度校验、枚举值校验、空值校验，再进入 `business`
- 对金额、状态流转、权限范围、业务归属等业务不变量，必须在 `domain` 或 `validator` 再做一次业务校验
- 禁止把原始前端字段直接透传到动态 SQL、脚本执行、模板渲染、文件路径拼接中

### 21.2 上传安全

- 上传文件必须校验文件大小、扩展名、MIME、业务白名单，不允许仅凭前端限制
- 文件名必须重新生成，不直接使用用户原始文件名作为存储键
- 上传目录、对象存储路径、临时文件目录必须有明确边界，禁止由外部直接控制完整路径
- 涉及图片、Excel、压缩包、富文本附件时，要明确是否需要做病毒扫描、内容审核、文件解压限制
- 禁止把上传文件落到可直接执行脚本的目录

### 21.3 下载与导出安全

- 下载接口必须校验权限和数据范围，不能“知道文件地址就能下”
- 导出接口必须复用列表查询同一套权限与数据范围约束
- 导出大文件时要控制行数、频率、并发，避免被滥用拖垮系统
- 文件下载响应头必须避免浏览器误判为可执行内容

### 21.4 回调与重放保护

- 支付回调、Webhook、第三方通知必须做签名校验、时间窗口校验、来源校验、幂等校验
- 回调原文、签名串、验签结果、处理结果必须可审计，但日志里要做脱敏
- 对重复回调、乱序回调、回放攻击必须有明确处理策略
- 回调接口不能只靠“路径足够隐蔽”来当作安全措施

### 21.5 SSRF、路径与注入风险

- 服务端主动发起 URL 请求时，必须校验协议、域名白名单、端口范围，避免 SSRF
- 禁止把用户输入直接拼成本地路径、远程 URL、Shell 命令、SQL 片段、排序字段
- 排序字段、筛选字段、导出字段必须走白名单映射
- 多表复杂查询统一放 XML 或结构化 Query 构造，不要散写字符串 SQL

## 22. 敏感数据治理

### 22.1 分类分级

- 团队至少要识别以下数据类型：账号身份数据、联系方式、证件数据、支付数据、凭证数据、业务机密数据
- 每类数据都要明确“可见范围、存储方式、日志策略、导出策略、脱敏规则”
- 涉及个人信息、财务数据、商户密钥的数据表和接口，评审时必须单独检查

### 22.2 存储与传输

- 敏感字段是否明文存储、密文存储、摘要存储，必须有明确规则，不能由开发随意决定
- 能用摘要满足校验需求时，不要存可逆明文
- 内外部传输都要明确是否需要 TLS、签名、加密或字段脱敏
- 缓存中存放敏感数据时，要明确过期时间、命名空间和清理策略

### 22.3 日志与审计

- 日志默认不打印完整请求体和完整响应体
- 对登录、权限变更、资金动作、导出动作、敏感字段查看动作必须写审计日志
- 审计日志至少记录操作人、对象、动作、结果、原因、`TraceId`
- 敏感字段明文查看必须单独授权并留痕

## 23. 供应链与发布安全

### 23.1 依赖治理

- 新引入第三方库、SDK、Starter 必须说明用途、维护状态、替代成本、潜在安全风险
- 禁止长期保留无维护来源或来路不明的 Jar 包
- 依赖升级要关注兼容性与已知漏洞，不要只看“能不能跑起来”
- 对支付、短信、对象存储、OCR、AI 等外部 SDK，要评估超时、重试、降级、签名、异常映射能力

### 23.2 构建与制品

- 构建过程必须可复现，禁止依赖个人电脑手工替换 Jar、手工改配置
- 镜像、部署包、配置文件中禁止包含生产密钥、测试后门、临时调试开关
- 发布前必须检查是否误提交 `.env`、测试账号、调试 SQL、临时脚本

### 23.3 上线前安全检查

- 是否存在硬编码密钥、密码、Token、回调密钥
- 是否存在未脱敏的敏感日志
- 是否存在未加权限控制的下载、导出、明文查看接口
- 是否存在可被重放的回调处理链路
- 是否存在直接拼接路径、URL、SQL、排序字段的代码
- 是否存在只在前端做校验、后端未兜底的关键参数

## 24. 与总清单的关系

- 安全检查项统一收口到 [templates-and-checklists.md](templates-and-checklists.md) 中的交付与上线总清单
- 本文件负责定义安全规则本身，不再额外维护重复版安全自检清单
- 需要新增安全检查项时，优先更新总清单，再按需回写本文件对应规则章节
