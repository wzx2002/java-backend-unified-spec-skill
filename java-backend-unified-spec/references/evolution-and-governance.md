# Java 后端演进与治理规范

## 25. 规则等级说明

- `必须`：默认要求，尤其适用于新项目、核心模块、对外契约变更、资金链路和高风险改动
- `推荐`：优先采用，老项目可结合成本与改造窗口逐步补齐
- `可选`：仅在系统复杂度、合规要求或团队协作成本达到阈值时启用

## 26. 兼容性与演进策略

### 26.1 对外契约稳定性

- 已发布接口、MQ 消息体、回调协议、导出模板、第三方对接字段都视为对外契约
- 修改对外契约前，必须明确影响范围、灰度方案、兼容期、通知对象
- 能追加字段解决的问题，不要优先选择破坏式改名或改语义
- 删除字段、改枚举语义、改状态机流转、改签名算法时，必须先做兼容方案

### 26.2 废弃策略

- 需要下线的接口、字段、枚举项、配置项要先标记废弃，再安排移除
- 废弃至少要说明：废弃原因、替代方案、预计移除时间、影响调用方
- 兼容期内要同时维护旧逻辑和新逻辑的边界，不要出现“文档说兼容，代码里其实已经删了”
- 对重要废弃项，建议在日志、监控或响应头中提供可观测信号

### 26.3 数据与状态演进

- 涉及历史数据迁移、状态枚举扩展、索引调整、字段改名时，必须先设计迁移顺序与回滚方案
- 禁止把“代码发布”和“数据修复”混成一步且没有兜底
- 影响资金、库存、额度、权限的数据迁移，必须有对账或校验脚本

## 27. 模块 README 与 ADR

### 27.1 模块 README

重要业务模块 `必须` 提供最小 README，普通支撑模块 `推荐` 提供最小 README。

README 至少包含：

- 模块目标
- 入口列表
- 核心用例
- 关键表或关键聚合
- 权限、数据范围、事务、幂等、并发策略
- 第三方依赖
- 回归测试范围

模块 README `推荐` 放在业务模块根目录或仓库 `docs/` 下统一管理，但路径必须统一。
可直接复用 [templates-and-checklists.md](templates-and-checklists.md) 中的模板。

### 27.2 ADR

以下场景 `必须` 补 ADR（Architecture Decision Record）：

- 顶层模块拆分方式调整
- 并发控制方案切换
- 权限模型调整
- 核心第三方接入方案选型
- 资金链路一致性方案调整
- 接口兼容性策略调整

ADR 至少说明：

- 背景
- 决策内容
- 备选方案
- 取舍理由
- 影响范围
- 后续约束

普通局部重构、小范围命名调整、无跨模块影响的实现替换，`可选` 是否补 ADR。

## 28. 静态检查与自动化门禁

### 28.1 最低门禁要求

- 代码格式化 `必须`
- 构建成功校验 `必须`
- 关键配置文件合法性校验 `必须`
- 单元测试或模块级关键链路测试 `必须`
- Checkstyle / Spotless / PMD / Sonar 中至少一类静态检查 `推荐`

### 28.2 推荐检查项

- 禁止新增 `Constants`、`CommonConstants` 这类兜底命名
- 禁止新增无注释的关键状态字段、金额字段、权限字段
- 禁止在业务代码中散写魔法字符串、魔法数字
- 禁止在 `Controller` 中写大段业务编排
- 禁止出现循环内查库导致的明显 N+1
- 禁止敏感信息直接打印日志

### 28.3 自动化校验思路

- PR 或提测前自动执行基础静态检查和测试
- 对目录命名、类命名、常量命名、枚举命名做规则扫描
- 对敏感日志关键字、硬编码密钥模式、危险配置开关做规则扫描
- 对模块 README、ADR、回归清单这类治理材料，按适用范围做“是否存在”的门禁校验

## 29. 评审与规范治理

### 29.1 评审关注点

代码评审 `必须` 至少检查：

1. 分层边界是否清晰。
2. 命名、枚举、常量分类是否符合规范。
3. 关键类、方法、字段注释是否完整。
4. 前置校验与业务断言边界是否清晰，简单守卫是否统一使用项目级 `Validate` helper，需要稳定业务错误码的失败语义是否统一使用 `BizException` / `BizAssert`。
5. 权限、事务、幂等、并发、审计是否补齐。
6. 是否引入了新的兼容性风险或安全风险。

### 29.2 规范维护方式

- 新规则优先更新到 skill 文档，不要长期停留在聊天记录、群消息、口头约定
- 当团队多次在同一问题上反复返工时，要把问题抽象成明确规则并补进文档
- 文档变更后，评审模板、脚手架、静态检查规则应尽量同步

### 29.3 AI 协作治理

- AI 生成代码后，`必须` 按“命名、注释、边界、安全、异常、事务、幂等、回归”维度自检
- AI 修改已有代码时，`必须` 同步判断是否需要更新注释、README、ADR、回归清单；仅在对应材料适用时强制更新
- 对高风险链路，AI 生成代码 `推荐` 附带关键约束说明，避免后续维护者只能靠猜
- 可直接复用 [templates-and-checklists.md](templates-and-checklists.md) 中的 PR 自检清单与评审清单
